# STM32-FreeRTOS 渐进式学习项目 Skill

## When to use
- 在本项目 `FreeRTOSDemo` 中进行STM32-FreeRTOS学习时
- 用户请求创建新的学习示例、修改代码或编写学习文档时
- 需要按照渐进式学习思路组织内容时
- 需要重点研究堆内存管理、流缓冲区、软件定时器模块时

## When NOT to use
- 在其他非学习项目中修改FreeRTOS代码时
- 进行生产环境项目开发时
- 需要完整移植FreeRTOS到新产品时

## 本项目工作原则

1. **渐进式学习**：从简单到复杂，每个学习单元只聚焦一个知识点，逐步深入
   - 每次只增加一个新概念/新API
   - 确保前一个概念理解后再进入下一个
   - 保持每个示例简洁，突出重点

2. **文档代码分离**：
   - 文档放在 `doc/` 目录下，包括学习笔记、经验沉淀
   - 示例代码放在 `projects/` 目录下，按学习阶段组织
   - 每个示例都有对应的文档说明学习要点

3. **经验沉淀**：
   - 学习过程中遇到的问题和解决方法沉淀到 `doc/experiences/` 按主题分类
   - stm32相关放到 `doc/experiences/stm32/`
   - FreeRTOS核心概念放到 `doc/experiences/freertos/`
   - 构建系统相关放到 `doc/experiences/build-system/`

4. **重点突出**：本项目重点研究三个模块：
   - **堆内存管理**：不同内存分配算法的实现和对比
   - **流缓冲区**：使用场景和实现原理
   - **软件定时器**：工作原理和使用注意事项

- 每个学习单元包含源代码和对应的README文档

## 命名规范

1. **目录命名**：
   - 使用小写字母和连字符，如 `01-getting-started`
   - 保持编号两位数字，便于排序
   - 描述性名称，清晰表达目录内容

2. **文件命名**：
   - 源文件：小写字母加下划线，如 `main.c`, `freertos_config.h`
   - 文档文件：使用中文命名，如 `README.md`, "学习笔记.md"
   - 避免空格和特殊字符

3. **代码命名**：
   - 遵循FreeRTOS原生命名规范
   - 函数名使用大驼峰或FreeRTOS风格 `vFunctionName`, `xFunctionName`
   - 变量名使用小写加下划线

## 文档编写要求

1. **每个学习单元必须包含 README.md**：
   - 开头说明本章节学习的核心知识点
   - 列出学习目标
   - 说明代码结构
   - 记录遇到的问题和解决方法

2. **经验沉淀文档要求**：
   - 每个主题一个目录，放置相关文档
   - 文档开头说明解决的问题
   - 提供清晰的解决步骤
   - 保留思考过程，不只是最终答案

3. **进度文档更新**：
   - `plan/当前进度.md` 记录当前学到哪个章节
   - 下次会话自动从当前进度继续

4. **文档使用中文编写**：便于理解和交流

## 代码创建要求

1. **渐进式增量开发**：
   - 每个示例只比前一个示例增加一个新知识点
   - 不要一次性引入多个新概念
   - 保持代码简洁，突出学习重点

2. **注释要求**：
   - 关键步骤添加详细注释说明
   - 对新使用的FreeRTOS API添加说明
   - 在关键点标注学习思考

3. **配置要求**：
   - 保持FreeRTOS配置简洁，只启用当前需要的功能
   - 针对STM32F103/STM32F4系列进行配置
   - 注释说明每个配置项的作用

4. **重点模块要求**：
   - 在堆内存管理相关示例中，添加对分配算法的注释说明
   - 在流缓冲区示例中，展示不同使用场景（中断与任务通信、大数据传输）
   - 在软件定时器示例中，演示不同精度和回调处理方式

## 针对"不太懂虚拟机和makefile"的渐进式引导

用户明确表示不太懂虚拟机和makefile，需要按照以下方式引导：

1. **初始阶段**：
   - 使用STM32CubeIDE创建项目，不要强求用户使用makefile
   - 接受CubeIDE生成的项目结构
   - 只关注FreeRTOS内核学习，不要深入构建系统细节

2. **遇到构建问题时**：
   - 提供清晰的操作步骤，而不是要求用户自己解决makefile问题
   - 建议使用CubeIDE的图形化配置界面
   - 如果需要修改配置，指导在哪里点击修改

3. **渐进式引入**：
   - 当用户对FreeRTOS有一定理解后，再逐步引入makefile概念
   - 不要一开始就要求用户理解复杂的构建系统
   - 保持学习曲线平缓

4. **环境问题处理**：
   - 如果用户遇到环境问题，优先提供CubeIDE解决方案
   - 不要强制要求使用QEMU或虚拟机
   - 优先在硬件或CubeIDE仿真环境中学习

5. **问题排查**：
   - 引导用户一步步排查问题，先检查CubeIDE配置
   - 再检查代码错误，最后才考虑构建系统问题
   - 保持耐心，循序渐进

## 工作流程

1. **开始会话**：
   1. 首先读取 `plan/当前进度.md` 了解当前学习进度
   2. 读取对应章节的文档，理解当前学习目标
   3. 根据用户需求，按照渐进式原则提供帮助

2. **创建新示例**：
   1. 在对应章节目录下创建新的学习单元目录
   2. 创建源代码文件，遵循命名规范
   3. 创建README.md文档说明学习要点
   4. 更新 `plan/当前进度.md`

3. **解答问题**：
   1. 优先参考 `doc/experiences/` 中已有的经验沉淀
   2. 如果没有，解答后将新经验沉淀到对应目录
   3. 针对用户对虚拟机和makefile不熟悉的情况，提供简单直接的解决方案

4. **重点模块处理**：
   - 当涉及堆内存管理、流缓冲区、软件定时器时，提供更详细的讲解
   - 添加更多注释说明实现原理
   - 对比不同实现方式的优缺点

## 示例

用户："帮我创建一个软件定时器的简单示例"

AI应该：
1. 检查当前进度，确定在 `04-software-timers` 目录下
2. 创建一个简单示例，只包含一个或两个软件定时器
3. 添加详细注释说明工作原理
4. 更新README说明学习要点
5. 不要一开始就引入复杂的定时器回调链或高精度场景

## 故障排查

- 如果忘记了项目结构，重新阅读本skill
- 如果不确定用户理解程度，保持渐进式，宁慢勿快
- 如果遇到构建问题，回到CubeIDE方案，不要坚持makefile
- 如果用户问了超出当前进度的问题，引导用户按顺序学习
